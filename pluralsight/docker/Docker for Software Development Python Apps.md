# Docker for Software Development Python Apps
A containerised Python application packages the application code, runtime, libraries, and supporting configuration into a repeatable unit. Docker gives a Flask service a consistent runtime from a developer workstation to production, reduces dependency conflicts, starts faster than a virtual machine, scales horizontally through container orchestration, and fits continuous integration and continuous deployment pipelines.

The sample application is a RESTful Flask service for a coffee shop. It manages coffees with an id, name, and version. The early version stores coffees in an in-memory list so the web service logic can be understood before database persistence is added. Flask routes connect HTTP methods and paths to Python functions. `GET /coffees` returns the full list, `GET /coffee/<id>` returns one coffee, `POST /coffee` creates a coffee, `PUT /coffee/<id>` updates a coffee, and `DELETE /coffee/<id>` deletes one.

The request handlers validate input and return clear HTTP responses. The list and read operations serialise Python dictionaries or model objects with Flask's `jsonify` function. Create and update operations read JSON from the request body. Delete operations remove the selected record and return a confirmation message. Missing data should produce `400 Bad Request`. A missing coffee should produce `404 Not Found`. A successful creation should return `201 Created`. A stale update uses the entity tag from the retrieved resource and compares it with the request's `If-Match` header before accepting the update. Production APIs commonly use `412 Precondition Failed` for a failed conditional request. The sample can return `409 Conflict` if that is the chosen API contract. Successful responses should include JSON and useful headers such as `Content-Type`, `Location`, and `ETag`.
## Docker images and containers
A Dockerfile defines how Docker builds an image. A Docker image is a read-only template. A container is a running instance of that image. The workflow is therefore Dockerfile, image, then container.

A Python web application usually starts from the official `python` image rather than from an empty image. Official images provide maintained base images for common operating systems, runtimes, databases, and services. A Dockerfile for the Flask service should set a working directory, copy `requirements.txt`, install dependencies with `pip install -r requirements.txt`, copy the application source, and start the app with a command such as `python ./app.py`.

Docker builds images in layers. Stable instructions should come earlier in the Dockerfile so Docker can reuse cached layers. Dependency installation usually changes less often than source code, so copying `requirements.txt` and installing packages should happen before copying the rest of the application. This keeps rebuilds faster during development.

A container runs with `docker run`. The `-d` option runs it in the background. A port mapping such as `-p 5000:5000` maps a host port to the container port. Flask must listen on `0.0.0.0` inside the container so traffic can reach it from outside the container. This setting exposes the development server to other interfaces, but Flask's development server and debugger should not run in production. Production deployments should use a production WSGI server such as Gunicorn, Waitress, uWSGI, or another supported deployment stack.

The image can be built with `docker build -t coffeeservice:1.0 .`. If no tag is supplied, Docker uses `latest`, but explicit tags are clearer for releases and rollbacks. `docker images` lists local images and `docker run` creates a container from one image. `docker ps` shows running containers. `docker exec -it <container-id> bash` opens an interactive shell inside a running container. The option is `-it`, which combines interactive input and a pseudo-TTY. `docker stop <container-id>` stops the container. `docker ps -a` shows stopped containers as well as running containers.
## Virtual environments inside containers
A Python virtual environment can still be useful inside a container. It separates application dependencies from system Python packages that the base image or operating system may need. It also keeps the container runtime closer to the local development environment.

The virtual environment should be created with a command such as `python -m venv /opt/venv`. Dependencies should be installed after the virtual environment path is active, otherwise packages may install into the image's system Python instead of the application environment. A Dockerfile should not rely on `RUN /opt/venv/bin/activate` to persist activation for later instructions, because each `RUN` executes in its own shell. The robust pattern is to set `ENV PATH="/opt/venv/bin:$PATH"`, then install packages and run Python through that environment. Another safe pattern is to call `/opt/venv/bin/pip` and `/opt/venv/bin/python` explicitly.
## Database persistence with SQLAlchemy
The service later replaces the in-memory coffee list with a relational database. SQLAlchemy supplies the Python SQL toolkit and object relational mapper, while Flask-SQLAlchemy simplifies integration with Flask. The coffee model maps a Python class to a database table with columns for id, name, and version. Class methods retrieve one coffee or all coffees. Instance methods save updates and delete records through the SQLAlchemy session.

The Flask application reads the database connection string from a `DATABASE_URL` environment variable. Local tests can default to an in-memory SQLite database. The containerised application can connect to MySQL with a database URL supplied at runtime. A production-grade URL should specify the appropriate SQLAlchemy dialect and driver for the installed MySQL driver, such as a `mysql+pymysql://...` style URL when PyMySQL is used.

A MySQL container can be built from the official MySQL image. The sample database image sets an initial root password and creates a database named `coffees`. The application image should not hard-code production credentials. Local credentials in teaching examples should be treated as placeholders only. An initialisation script copied to `/docker-entrypoint-initdb.d` can create the `coffees` database and the `coffee` table when the container starts. During local Docker Desktop development, `host.docker.internal` can let a container reach services on the host. On native Linux Docker Engine, the name may require configuration such as `--add-host=host.docker.internal:host-gateway`, a Compose equivalent, or a user-defined network with service names. In production, orchestration systems such as Docker Compose or Kubernetes should provide stable service names rather than relying on the host address.
## Testing inside Docker
Running tests inside Docker gives developers a repeatable test environment. The tests use pytest and Flask's test client. The test suite preloads known coffee records into an in-memory SQLite database before each test and cleans them up afterwards. Test cases cover listing coffees, retrieving one coffee, handling a missing coffee, creating a coffee, updating a coffee, rejecting stale updates, handling missing records on update, deleting a coffee, and rejecting deletion of a missing record.

Pytest fixtures manage setup and cleanup. A function-scoped fixture runs around each test. It can insert test data before the test executes, yield that data to the test, then remove the data after the test finishes. Wider fixture scopes, such as class, module, package, and session, suit resources that should be created less often.

Tests can run with `python -m pytest`, either against a named test file or through pytest discovery. A failure should stop the image build and show the failing assertion so the error is fixed before the image is published. A multi-stage Dockerfile can make tests part of the build. One stage prepares the application and runs pytest. If any test fails, the Docker build fails and no deployable image is produced. A later production stage can start from a tested stage or copy only required artefacts from it. Multi-stage builds also allow the final image to omit development-only files when those files are not needed at runtime.
## Linting inside Docker
A linter performs static analysis before code reaches production. PyFlakes focuses on programming errors. Flake8 combines PyFlakes, pycodestyle, and McCabe complexity checks. Ruff is a fast Rust-based linter and formatter. Pylint is comprehensive, configurable, and widely used in Python projects.

Pylint reports errors, warnings, refactor suggestions, and convention issues. Errors identify serious problems. Warnings identify likely defects or risky code. Refactor messages highlight complexity or structure problems. Convention messages identify style violations.

The lint stage can use the same project setup as the test stage, including the working directory, virtual environment, installed requirements, and copied source. It can then run `pylint .` from the application directory. A lint stage can run before the test stage. If Pylint fails, the build stops before tests run. The Dockerfile can install Pylint only in the lint stage so it does not increase the final image size. A `pylintrc` file can disable checks that the team chooses not to enforce, such as missing docstring checks in a small teaching application.

Pylint can also identify useful fixes. Catching broad exceptions should be replaced with catching specific exceptions such as `SQLAlchemyError`. Variables named `id` should be renamed to avoid shadowing Python's built-in `id` function. Module filenames should use snake case, so `coffee.py` is better than `Coffee.py`. Standard library imports should come before third-party imports. Pytest fixture warnings can be handled with a plugin such as `pylint-pytest`, which teaches Pylint how pytest fixture injection works without disabling a useful warning globally.
## Publishing a containerised Python application
Before publishing a container image, the application should expose useful logs, accept external configuration, and handle sensitive information safely.

Python's standard logging module supports log levels such as critical, error, warning, info, and debug. Loggers form a hierarchy. Handlers send messages to destinations such as standard output, files, or network services. Formatters control the message structure. Containerised applications normally write logs to standard output and standard error so `docker logs` and orchestration logging systems can collect them.

A logging configuration file can define a root logger, a console handler, and a formatter that includes the timestamp, severity, logger name, and message. The application can load this file with `logging.config.fileConfig` and set `disable_existing_loggers=False` so framework loggers still work. Request handlers and persistence methods can then log events such as retrieving coffees, creating records, saving entities, and failed updates.

External configuration also keeps the same image usable across development, test, staging, and production. The image should define defaults only when they are safe and useful. Runtime configuration should override those defaults without rebuilding the image. Configuration should come from outside the image where practical. Environment variables suit simple values such as `DATABASE_URL`. The `docker run -e` option can override environment variables when a container starts. Bind mounts suit configuration files such as `logging.ini`. A bind mount such as `--mount type=bind,source=$PWD/config,target=/config` lets the container read configuration supplied by the runtime environment.

When build stages are reorganised to externalise configuration files, the final stage should still depend on the lint and test stages so quality checks run before the production image is produced. Copying tested application files from an earlier stage is one way to keep that dependency while avoiding unwanted configuration files in the final image. Sensitive values should not be stored in a Dockerfile, source code, or plain image layers. Passwords, certificates, tokens, and API keys should be passed through secret management mechanisms provided by the orchestration platform. Docker Compose can mount secrets under `/run/secrets`. Kubernetes can expose secrets as environment variables or mounted files. The application should read those values at runtime rather than baking them into the image.
